### Session 2 (GDT Main, ~11:45 AM) — Debugging & Root Cause Fix

**Problem:** Door notifications stopped working. Caleb reported door opening/closing with no notifications.

**Investigation:**
- Found **86 monitor processes** running on Mac Studio
- All processes stuck in "No route to host" error loops
- None actually connected to the Hue Bridge

**Root Cause Analysis:**
1. **launchd/cron environment issue**: Processes spawned by cron start before network routing is fully initialized
2. **Broken single-instance check**: `pgrep` pattern matched zombie processes, so cron kept spawning new ones
3. **No health monitoring**: No way to detect process exists but isn't working
4. **Cascade**: 86 processes accumulated over 7 hours, all failing

**Verification:**
- curl works from cron environment (network IS available)
- Python requests works from cron environment
- SSE endpoint works from cron environment
- Issue was timing + lack of pre-check + no health monitoring

**Fix Applied — New `ensure-running.sh`:**
1. **Network pre-check**: Verify bridge reachable via curl before starting (with 6 retries, 5s apart)
2. **PID file management**: Proper single-instance using PID file instead of pgrep
3. **Health monitoring**: Check if process is actually connected (looks for "Connected!" in recent logs)
4. **Auto-recovery**: Kill unhealthy processes and restart
5. **Better logging**: Diagnostics in `~/.hue-monitor/logs/ensure.log`

**Current State:**
- Single monitor process running (PID tracked in ~/.hue-monitor/monitor.pid)
- Connected to Hue Bridge ✅
- Cron checks health every 5 minutes
- Door events are **instant** (SSE push, <1 second total latency)

## Architecture Notes

**Event Flow (real-time):**
```
Door opens → Bridge detects → SSE push (~50ms) → Monitor receives → Telegram API (~200ms) → Notification
```

**Health Check Flow (every 5 min):**
```
Cron runs ensure-running.sh → Check PID file → Check process exists → Check recent logs for "Connected!" → If unhealthy: kill & restart with network pre-check
```

**Why launchd failed but SSH worked:**
- launchd spawns processes early in boot/login before network routes are fully populated
- "No route to host" means kernel can't find path to 192.168.100.122
- SSH sessions have full network stack available
- Fix: Pre-check network with curl retry loop before starting monitor
